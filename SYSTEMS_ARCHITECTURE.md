# Flip Runner: Technical Design & Systems Architecture
**Role:** Senior Lead Game Designer & Systems Architect
**Target:** Strict, Deterministic 2D Endless Runner

This document outlines the architectural blueprints, logical flows, and mathematical models for four core pillars of the "Flip Runner" technical design.

---

## 1. Telemetry and "Death Heatmapping"

To reliably differentiate between player skill issues and objective level design flaws (e.g., blind jumps, unfair hitboxes), telemetry must be highly structured and segmented by player cohort.

### Data Architecture & Payload
Instead of sending real-time streams (which causes server bloat), the client batches telemetry and sends a compressed payload at the end of a run or upon death. 

**Death Event JSON Schema:**
```json
{
  "event_type": "player_death",
  "session_id": "uuid",
  "player_cohort": "tier_3_advanced", // Based on lifetime progression
  "death_pos": {"x": 12450.5, "y": -42.0},
  "chunk_id": "chk_caves_04",
  "relative_chunk_x": 145.2,
  "state_at_death": {
    "velocity_x": 15.0,
    "velocity_y": -9.8,
    "jump_buffered": false,
    "last_input_time_ms": 120
  }
}
```

### Separating Design Flaws from Skill Issues
To identify design flaws, we implement **Cohort-Based Density Clustering (DBSCAN)** on the backend.
1. **The Faucet/Sink Telemetry:** Track every coin/gem generated (`faucet_amount`, `source`) vs. spent (`sink_amount`, `destination`). 
2. **The "Frustration Index":** If the bottom 20% of players die at `x: 145.2` in `chk_caves_04`, it's a skill check. If the **top 10%** of players (High Cohort) suddenly show a mortality rate > 15% at `x: 145.2`, it indicates a mechanical failure (e.g., a frame-perfect jump expectation or visual obfuscation).
3. **Actionable Heatmaps:** Heatmap visuals overlay `relative_chunk_x/y` onto the chunk's collision mesh. Red zones generated by High Cohort players are automatically flagged in Jira/Trello via backend hooks for immediate designer review.

## 2. Chunk Metadata and Sequencing Algorithms

An endless runner requires a "Director" to maintain flow state. If the game randomly stitches difficult chunks together, the player experiences mathematical unfairness. 

### Database Schema (Chunk Metadata)
Chunks are pre-authored and baked. Their metadata governs how they connect.
*   `ChunkID`: `String` (e.g., "volcano_hard_01")
*   `Difficulty`: `Float` (1.0 to 10.0)
*   `EntryHeight`: `Int` (e.g., 2 - Must match previous chunk's ExitHeight)
*   `ExitHeight`: `Int` (e.g., -1)
*   `MinSpeed`: `Float` (Required horizontal velocity to clear)
*   `IntensityPacing`: `Enum` (Rest, Build, Peak)

### The "Director Algorithm"
We use a **Sinusoidal Intensity Curve** combined with a **Token Bucket** to dynamically stitch chunks.

**The Math:**
```math
TargetDifficulty = BaseRamp(Distance) + Amplitude * sin(Frequency * Distance)
```
*   `BaseRamp`: Slowly increases over time (e.g., `1.0 + (Distance / 1000)`).
*   `Sinusoidal Wave`: Creates peaks (intense gameplay) and valleys (rest chunks) to prevent cognitive overload.

**Selection Logic (Real-Time):**
1. Evaluate `TargetDifficulty` for the next chunk sequence.
2. Query the chunk pool: `WHERE EntryHeight == CurrentExitHeight AND MinSpeed <= PlayerCurrentSpeed`.
3. Filter pool by `TargetDifficulty +/- 1.0`.
4. If the player just completed a "Peak" intensity chunk, forcibly select a "Rest" chunk to allow cognitive recovery and pacing variation.

## 3. Dynamic Economy Controllers

Hardcoding values in a LiveOps game is a critical failure point. Variables must be driven by a Remote Config State (e.g., Firebase Remote Config, PlayFab, or custom JSON from S3).

### Decoupling Architecture
*   **The Config State:** A master JSON file dictating: `BaseUpgradeCost`, `CostMultiplier`, `BaseXP`, `MaxLevel`, `TokenDropRate`.
*   **Client Implementation:** The client downloads this JSON on boot, caches it, and uses a Dependency Injected `EconomyService` to serve the values. 

### Calculating "Days to Max Level" (D2M)
To retain F2P (Free-to-Play) users while incentivizing monetization, we must balance Faucets (Daily Quests, Run Rewards) against Sinks (Upgrades).

```math
Total Cost (Level N) = BaseCost \times (CostMultiplier)^{N-1}
```

To find Days to Max (D2M):
```math
D2M = \frac{\sum_{i=1}^{MaxLevel} Cost(i)}{(DailySessionLength \times EarnRatePerMinute) + DailyQuestRewards}
```

**Implementation Strategy:**
If `D2M` > 45 days, player churn increases drastically. If `D2M` < 14 days, monetization cannibalization occurs (players max out before feeling the pinch to buy). 
The Remote Config allows us to tweak `CostMultiplier` and `DailyQuestRewards` to keep the median F2P player clamped between 21 and 30 days, adjusting live as data rolls in.

## 4. Asynchronous "Ghost" Racing

Ghost racing acts as a highly effective, low-friction multiplayer system that drives retention and monetization through cosmetic envy.

### Technical Implementation: Deterministic Input Recording
Storing 60 FPS X/Y transform data causes severe server bloat. Because Flip Runner is built on strict, deterministic physics, we **only record inputs**.
*   **Payload structure:** `Seed` (Random generation seed to ensure chunk identicality), `[Frame: 140, Input: Jump_Down], [Frame: 165, Input: Jump_Up]`.
*   **File Size:** A 5-minute run becomes < 2KB instead of several Megabytes.
*   **Playback:** The Ghost is an invisible simulated entity running the exact same physics frame-tick as the player, reading the input payload. The visual renderer simply interpolates the simulated Ghost's position to render the semi-transparent avatar.

### Matchmaking & Psychological Incentive
We do not match the player with a random ghost. We match them using a **"Carrot on a Stick" ELO system**.
*   **The Algorithm:** Query the backend for a ghost whose run time/score is exactly `102% to 108%` of the current player's Personal Best (PB). 
*   **Monetization Hook:** Weight the database query to prioritize Ghosts wearing Premium Cosmetics (Skins, Trail FX) that the current player does not own. 
*   **Result:** The player races a ghost they can visually see, who is only *slightly* better than them (making victory feel achievable), while simultaneously being advertised premium store items through gameplay rather than intrusive UI.
